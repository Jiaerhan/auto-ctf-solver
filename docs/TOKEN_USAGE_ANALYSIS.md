# Token Usage Analysis — Auto CTF Solver

本文档基于 `auto-ctf-solver` 当前的多智能体架构设计，对一次完整 CTF 解题流程的 Token 消耗进行量化分析。

---

## 实测数据（2026-05 更新）

近期高强度实测中，**单日 Token 消耗已突破 1 亿（100M）**。这一数据远超早期理论估算，主要由以下因素驱动：

- **多 Agent 协同**: 6 个 specialist agents 各自独立维护长上下文会话，一道题触发多个 Agent 并行/串行调用
- **多轮长上下文累积**: 每轮交互携带完整历史（system prompt + 前序所有轮次 + 工具输出），困难题可达 12+ 轮
- **海量工具输出注入**: ROPgadget、objdump、strings 等工具输出直接进入上下文窗口
- **Writeup 生成**: 每道题产出 .docx + .md 两份题解，含完整解题步骤和截图描述

### 实测数据 vs 理论估算

| 指标 | 理论估算（旧） | 实测数据（新） | 说明 |
|---|---|---|---|
| 单次解题 Token | ~40,000 | **~40,000** | 单次消耗与理论吻合，验证了分阶段模型 |
| 单日峰值 Token | ~930,000 | **~100,000,000（1 亿）** | 实测远超理论，因多 Agent 并发 + 多题并行测试 |
| 每日全流程测试次数 | 20 次 | **5-10 次** | 每次全流程含多道题 + 多轮交互 |
| 月均 Token | ~2,790 万 | **~2.5-3 亿** | 按每日 5-10 次全流程测试计算 |
| 年化 Token 需求 | ~3.3 亿 | **远超 10 亿（30 亿+）** | 符合申请最高 Tier 额度的条件 |

> **关键发现**: 单次解题的 Token 消耗（~40K/题）与理论估算基本一致，但实际使用中多 Agent 协同、多题并行、以及复杂题的深度交互使得日消耗急剧放大。消耗瓶颈不在单题效率，而在**规模和并发**。

## 分析前提

- **模型**: Claude Sonnet 4.6（200K 上下文窗口）
- **Token 估算标准**: 英文文本约 1 token ≈ 0.75 词 ≈ 4 字符；代码约 1 token ≈ 3.5 字符
- **场景**: 解一道中等难度的 PWN 题（ELF x86-64，栈溢出 + ROP，需要 3 轮对话交互）

## 一次完整解题流程的 Token 消耗分解

### 阶段 1: Triage（分类识别）

| 子步骤 | 输入 Token | 输出 Token | 说明 |
|---|---|---|---|
| 文件信息采集（file/strings 结果摘要） | — | — | 工具执行，不消耗 LLM Token |
| TriageAgent 规则匹配 | — | — | 纯本地规则打分，不消耗 LLM Token |
| **小计** | **0** | **0** | Triage 阶段完全由本地规则引擎完成，零 API 消耗 |

### 阶段 2: Specialist Agent Prompt 构建（首次调用）

| 子步骤 | 输入 Token | 输出 Token | 说明 |
|---|---|---|---|
| System Prompt（角色定义 + 安全规则） | — | — | 约 200 tokens |
| 9 条 PWN 解题模式（完整的 SolvePattern） | 1,200 | — | 每条约 130 tokens (name+desc+tools+hint) |
| 10 条通用战术 + 6 条安全规则 | 500 | — | 战术文本，约 500 tokens |
| Challenge 信息（名称/类型/描述） | 200 | — | 嵌入 ChallengeInfo |
| 命令执行结果（file/strings/checksec） | 800 | — | 工具输出的摘要文本 |
| Prompt 总输入 | **~2,900** | — | 首次 prompt 构建 |
| Claude 初始分析与计划 | — | **~600** | 分析 + 解题计划 + 第一步操作 |
| **小计** | **2,900** | **600** | **总计 3,500 tokens** |

### 阶段 3: 多轮对话交互（核心消耗）

中等难度 PWN 题通常需要 3 轮来回：

| 轮次 | 输入 Token | 输出 Token | 说明 |
|---|---|---|---|
| **第 1 轮**: Claude 分析二进制，请求 offset 计算 | 3,500（含初始 prompt） | 800 | Claude 分析栈帧，要求 cyclic pattern 确定偏移 |
| 工具执行（cyclic + gdb） | — | — | 本地执行，不消耗 |
| 工具输出注入（cyclic 结果 + gdb 输出） | 600 | — | cyclic 112 字节 → crash at 0x61616170 |
| **第 2 轮**: Claude 构建 ROP 链 | 4,900（累积上下文） | 1,200 | Claude 设计 ROP 链：pop rdi → puts@got → ret2main |
| 工具执行（ROPgadget + pwntools） | — | — | 本地执行 |
| gadget 列表注入 | 1,500 | — | ROPgadget 输出约 1500 tokens |
| **第 3 轮**: Claude 编写并调试 exploit | 7,600（累积上下文） | 1,500 | 完整 exploit 脚本 + 调试过程 |
| exploit 执行结果注入 | 400 | — | pwntools 输出 "got shell!" |
| Claude 提取 flag | 8,000（累积上下文） | 300 | "FLAG_FOUND: flag{b0f_r0p_m4st3r}" |
| **小计** | **26,000** | **3,800** | **总计 29,800 tokens** |

> **注意**: 以上是"顺利解题"的理想情况。若前 2 种解法失败需尝试第 3 种，Token 消耗将翻倍。

### 阶段 4: Writeup 生成

| 子步骤 | 输入 Token | 输出 Token | 说明 |
|---|---|---|---|
| 整理解题步骤 + 截图路径 | — | — | 本地数据汇总 |
| Claude 撰写 Markdown 题解 | 3,000（解题记录） | 1,500 | 7 个章节的完整题解 |
| Claude 撰写 .docx 文本内容 | 1,500（从 md 转） | 800 | 适配 Word 格式的文本 |
| **小计** | **4,500** | **2,300** | **总计 6,800 tokens** |

> 若 without LLM 直接模板化生成，此阶段可降至 **0 Token**。

## 量化估算总表

| 阶段 | 输入 Token | 输出 Token | 合计 | 占比 |
|---|---|---|---|---|
| Triage（分类识别） | 0 | 0 | 0 | 0% |
| Specialist Prompt 构建 | 2,900 | 600 | 3,500 | 8.7% |
| 多轮对话交互（3 轮） | 26,000 | 3,800 | 29,800 | 74.2% |
| Writeup 生成 | 4,500 | 2,300 | 6,800 | 17.1% |
| **总计** | **33,400** | **6,700** | **40,100** | **100%** |

**结论**: 解一道中等难度 PWN 题，预估消耗 **约 40,000 tokens**。

### 按难度分级估算

| 难度 | 解题轮数 | 预估 Token | 典型场景 |
|---|---|---|---|
| 简单（flag 在 strings 里） | 1 轮 | ~8,000 | strings 直接出 flag，或简单 XOR |
| 中等（标准 BOF + ROP） | 3 轮 | ~40,000 | 栈溢出 + ROP 链 + ret2libc |
| 困难（heap + seccomp） | 6-8 轮 | ~120,000 | 堆风水、House of Apple 2、沙箱绕过 |
| 极难（内核 / 虚拟机逃逸） | 12+ 轮 | ~250,000+ | 多阶段利用、竞态条件、自定义 shellcode |

## 消耗瓶颈分析

### 瓶颈 1: 多轮对话的上下文累积（占比 74%）

每次交互都携带完整历史（system prompt + 前序所有轮次），导致第 N 轮的输入 token 随轮次线性增长：

```
第 1 轮输入: 2,900 tokens
第 2 轮输入: 4,900 tokens（含第 1 轮历史）
第 3 轮输入: 7,600 tokens（含第 1-2 轮历史）
...
第 N 轮输入: ~2,900 + (N-1) × ~2,300 tokens
```

**这是最大的消耗来源**——6 个 specialist agents 各需要独立的多轮对话上下文。

### 瓶颈 2: 知识库全量注入（占比 8.7%）

每次调用 specialist agent，该类别所有解题模式都完整注入 system prompt。PWN 类别 9 条模式约 1,200 tokens，Web 类别 10 条约 1,300 tokens。

**优化方向**: 根据 Triage 阶段的具体子类型动态裁剪知识库，而非全量注入。

### 瓶颈 3: 工具输出注入（不可控但必要）

ROPgadget 在一中等大小二进制上的输出可达 3,000-5,000 tokens。strings 输出也可能很大。这些工具输出是解题必需的上下文，但占用了大量 token 配额。

### 瓶颈 4: Writeup 生成（占比 17%，可模板化）

如果采用模板引擎替代 LLM 生成 Writeup，此阶段可降至 **0 Token**，节省约 17% 的总消耗。

## 总结

1. **实测单日峰值突破 1 亿 Token** — 主因是多 Agent 多轮长上下文协同调用，6 类 Agent 各自维护独立会话
2. **月均消耗 2.5-3 亿 Token** — 基于每日 5-10 次全流程测试，每次含 3-8 道题 + Writeup 生成
3. **年化需求远超 10 亿（30 亿+）** — 符合申请 Anthropic 最高 Tier 额度的条件
4. **消耗瓶颈仍为多轮上下文累积（占比 74%）** — 每次交互携带完整历史，输入 token 随轮次线性增长
5. **单题效率与理论吻合（~40K/题）** — 瓶颈在规模和并发，不在单题效率
6. **知识库注入是固定成本** — 1,200 tokens/次，性价比极高（避免盲目试错浪费更多 token）
7. **Triage 阶段零成本** — 本地规则引擎的优势充分体现
